# ERD Diagram Update: Schema 7.22 → 7.24

**Date:** 2026-07-09  
**Status:** Approved

## Goal

Copy the five 7.22 ER diagram SVGs and produce identical-layout 7.24 versions reflecting the current `schema.mermaid` schema. Update `index.md` to reference the new files.

## SVG Files

Source: `static/img/documentation/user-guide/process-engine/database/erd_722_{bpmn,dmn,cmmn,history,identity}.svg`  
Target: same directory, renamed to `erd_724_*.svg`

## Technical Context

The SVGs are Inkscape-generated with all text stored as `<use xlink:href="#glyphX-N">` elements — not `<text>` nodes. Each character is a pre-defined glyph path. Each SVG has its own independent glyph set covering only the characters used in that file.

## Implementation Plan

### Step 1 — Glyph decoder

Write a Python script `scripts/erd_update.py`. For each SVG:

1. Parse glyph definitions: each `<g id="glyphF-N">` contains a `<path d="...">`.
2. Build a glyph-path fingerprint map: `path_data → character`.  
   Strategy: use known strings (table names follow `ACT_[A-Z_]+` pattern; the first line in each table block is the table name, which is identifiable from the mermaid schema) to reverse-map glyph indices to ASCII characters.
3. Walk `<use>` elements grouped by y-coordinate (same y ≈ same text line); reconstruct text per line.
4. Output: `{table_name: [col_name, ...]}` per SVG.

### Step 2 — Schema diff

Parse `schema.mermaid`:
- Extract table names and column names per table (ignore column types and PK/FK markers for the SVG column text; the SVGs show only the column name).
- Build same structure: `{table_name: [col_name, ...]}`.
- Diff per table: `added = 7.24_cols - 7.22_cols`, `removed = 7.22_cols - 7.24_cols`.

### Step 3 — SVG patcher

For each SVG, for each changed table in that SVG:

**Remove column row:**
- Find all `<use>` elements whose text decodes to the removed column name (same y-coordinate group).
- Delete them.
- Shift all `<use>`, `<rect>`, and line elements with y > removed_row_y upward by row_height (~13pt).
- Shrink the table bounding `<rect>` height by row_height.

**Add column row:**
- Determine insert position (alphabetical order or end of column list).
- Clone `<use>` elements from an existing column row; update glyph references to spell the new column name.
- Shift all elements below insert_y downward by row_height.
- Grow the table bounding `<rect>` height by row_height.

**Glyph availability:**
- Before encoding a new column name, check that all required characters exist in the SVG's glyph set.
- If a character is missing, locate it in any other SVG's glyph set, copy the `<g id="glyphX-N">` definition into the target SVG with a new ID, and use it.

### Step 4 — Output and wiring

- Write output SVGs as `erd_724_*.svg` (do not overwrite originals).
- In `docs/documentation/user-guide/process-engine/database/database-schema/index.md`, replace all `erd_722_` references with `erd_724_`.

## Scope Boundaries

- Layout (table positions, relationship lines) is **not changed**.
- Column types, PK/FK markers in SVG column text: preserve whatever the 7.22 SVG shows for unchanged columns; new columns follow the same text format as adjacent columns.
- The script is a one-shot tool; no need to make it a general ERD generator.

## Success Criteria

1. `erd_724_*.svg` files render correctly in browser.
2. All tables in the 7.24 SVGs match `schema.mermaid` column lists.
3. `index.md` references point to the new files.
4. No 7.22-only columns remain; no 7.24 column is missing.
